# 点赞系统与个人中心点赞统计设计

## 1. 目标

为文章增加真实可用的点赞系统，并将个人中心顶部“点赞”计数器改为“当前用户发布的所有文章累计获得的点赞数”。

本次实现范围包含：

- 用户可以在文章详情页点赞与取消点赞
- 同一用户对同一篇文章最多只能保留一次点赞
- 作者不能给自己的文章点赞
- 文章详情返回真实点赞状态与点赞数
- 个人中心顶部点赞统计改为作者获赞总数

本次不包含：

- 文章列表页点赞按钮
- 点赞通知
- “我点赞过的文章”页面
- 点赞排行榜

## 2. 当前现状

- 现有系统只有收藏能力，数据表为 `favorites`
- 文章详情页只有收藏按钮，没有点赞按钮
- 个人中心顶部的 `stats.likeCount` 目前没有真实点赞来源
- 文章实体 `Article` 目前没有 `likeCount`

## 3. 方案选择

### 方案 A：新增独立 `likes` 表并维护文章点赞计数

这是本次采用的方案。

优点：

- 语义清晰，和收藏系统边界明确
- 易于保证“每个用户每篇文章只能点赞一次”
- 支持取消点赞
- 便于后续扩展“我点赞过的文章”“点赞通知”等功能

缺点：

- 需要新增数据表和对应仓储、服务逻辑

### 方案 B：只在 `articles` 表中直接增减点赞数

不采用。

原因：

- 无法记录是谁点的赞
- 不能保证单用户单文章唯一点赞
- 无法支持取消点赞

### 方案 C：复用 `favorites` 表并增加类型字段

不采用。

原因：

- 收藏与点赞语义不同
- 后续查询、维护和统计会混乱

## 4. 数据模型设计

### 4.1 新增 `likes` 表

字段：

- `id`
- `user_id`
- `article_id`
- `created_at`

约束：

- `user_id` 外键关联 `users.id`
- `article_id` 外键关联 `articles.id`
- 唯一约束：`(user_id, article_id)`

### 4.2 修改 `articles` 表

新增字段：

- `like_count bigint not null default 0`

用途：

- 作为文章当前点赞总数缓存字段
- 点赞与取消点赞时同步增减

## 5. 业务规则

- 用户必须登录后才能点赞
- 用户可以点赞别人的文章
- 作者不能给自己的文章点赞
- 同一用户同一文章只能点赞一次
- 用户可以取消点赞
- 取消点赞后文章点赞数减一，且不能减到负数
- 只有文章存在且用户有权访问该文章时，才允许进行点赞相关操作

## 6. 后端接口设计

### 6.1 新增点赞接口

`POST /api/v1/articles/{articleId}/like`

作用：

- 当前登录用户为文章点赞

成功返回：

- 当前文章最新的点赞数
- 当前文章点赞状态 `liked = true`

失败场景：

- 未登录
- 文章不存在
- 作者给自己文章点赞
- 重复点赞

### 6.2 新增取消点赞接口

`DELETE /api/v1/articles/{articleId}/like`

作用：

- 当前登录用户取消对文章的点赞

成功返回：

- 当前文章最新的点赞数
- 当前文章点赞状态 `liked = false`

失败场景：

- 未登录
- 文章不存在
- 当前用户未点赞该文章

## 7. 返回结构调整

### 7.1 文章摘要与文章详情

以下 DTO 同步增加字段：

- `likeCount`
- `liked`

涉及对象：

- `ArticleSummaryResponse`
- `ArticleDetailResponse`

说明：

- `likeCount` 表示文章当前总点赞数
- `liked` 表示当前登录用户是否已点赞该文章

### 7.2 用户资料统计

`UserProfileResponse.stats.likeCount` 改为真实统计值：

- 当前用户作为作者，其所有文章累计获得的点赞总数

统计口径：

- 统计作者全部文章收到的点赞总数
- 与用户自己点过多少赞无关
- 与收藏数统计分离

## 8. 服务与仓储设计

### 8.1 新增实体与仓储

新增：

- `Like` 实体
- `LikeRepository`

仓储能力至少包含：

- 按用户和文章判断是否已点赞
- 按用户和文章查询点赞记录
- 按文章统计点赞数
- 按作者统计获赞总数

### 8.2 点赞服务

建议新增独立服务，例如 `ArticleLikeService`，职责仅包含：

- 点赞
- 取消点赞
- 点赞状态查询

文章服务 `ArticleService` 继续负责文章详情、列表与作者文章查询，但会复用点赞查询结果填充 `likeCount` 与 `liked`。

## 9. 前端交互设计

### 9.1 文章详情页

仅在 `ArticleDetail.vue` 中新增点赞按钮。

交互规则：

- 未点赞：显示“点赞（x）”
- 已点赞：显示“取消点赞（x）”
- 作者本人：按钮禁用，并显示“不能给自己的文章点赞”
- 点赞成功：按钮状态切换为已点赞，计数立即加一
- 取消点赞成功：按钮状态切换为未点赞，计数立即减一

### 9.2 个人中心

`Profile.vue` 顶部统计区中的“点赞”数值，改为读取后端返回的真实 `stats.likeCount`。

无需前端自行汇总用户文章分页数据。

## 10. 测试范围

### 10.1 后端测试

覆盖以下场景：

- 用户可以给别人的文章点赞
- 同一用户重复点赞被拒绝
- 用户可以取消点赞
- 作者不能给自己的文章点赞
- 点赞后文章详情返回 `liked = true`
- 取消点赞后文章详情返回 `liked = false`
- 用户资料返回的 `stats.likeCount` 为作者获赞总数

### 10.2 前端测试

覆盖以下场景：

- 文章详情页存在点赞按钮
- 未点赞与已点赞文案正确切换
- 作者本人点赞按钮禁用
- 点赞后界面计数即时更新
- 取消点赞后界面计数即时回退
- 个人中心继续展示后端返回的真实点赞统计

## 11. 风险与约束

- 如果数据库尚未执行 `like_count` 字段与 `likes` 表迁移，后端将无法正常运行该功能
- 点赞数采用“写时更新 articles.like_count”的方式，后续若引入高并发场景，需要进一步评估并发一致性
- 当前阶段不做消息通知与反作弊逻辑，后续如需上线生产，应补充限流、审计和异常行为监控

## 12. 实施顺序

推荐顺序：

1. 新增数据库结构与实体仓储
2. 编写后端点赞与统计失败测试
3. 实现后端点赞逻辑与资料统计修正
4. 编写前端失败测试
5. 实现文章详情页点赞按钮与状态更新
6. 回归验证个人中心点赞统计
