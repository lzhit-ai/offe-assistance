# 个人中心头像上传与昵称修改设计

## 目标

在现有个人中心基础上新增两项能力：

- 用户上传并更换个人头像
- 用户修改展示昵称

同时明确区分两个概念：

- `username`：登录账号，唯一，不可修改
- `nickname`：展示昵称，唯一，可修改

## 范围

本次设计只覆盖以下内容：

- 数据模型扩展
- 个人资料查询返回字段调整
- 昵称修改接口
- 头像上传接口
- 个人中心页面交互改造
- 前后端校验和错误提示
- 对应的测试补充

本次不包含：

- 云存储接入
- 图片裁剪
- 图片审核
- 用户名修改
- 批量资料编辑

## 数据模型

基于现有 `users` 表保留以下字段语义：

- `username`：登录账号，唯一，不可修改
- `avatar`：头像访问地址，可为空

新增字段：

- `nickname`：展示昵称，唯一，可为空或在初始化时回填

建议后端展示逻辑统一采用：

- 优先返回 `nickname`
- 当 `nickname` 为空时，前端显示时回退到 `username`

## 昵称规则

昵称规则如下：

- 用户可修改
- 全站唯一
- 去除首尾空格后再参与校验
- 最大长度为 10 个中文字符
- 不允许为空字符串

实现时以后端校验为准，前端做即时校验提示。

说明：

- 由于中英文字符宽度不同，前端可先按“长度不超过 10”做直观限制
- 后端应以统一字符长度规则兜底，避免绕过前端校验

## 接口设计

### 1. 获取当前用户资料

继续复用现有接口：

- `GET /api/v1/auth/me`

返回字段补充或明确为：

- `id`
- `username`
- `nickname`
- `phone`
- `email`
- `avatar`
- `registerTime`
- `stats`

### 2. 修改昵称

新增接口：

- `PATCH /api/v1/users/me/profile`

请求体：

```json
{
  "nickname": "新的昵称"
}
```

行为约束：

- 仅修改 `nickname`
- 不允许修改 `username`
- 若昵称与当前用户已有昵称一致，直接返回成功
- 若昵称已被其他用户占用，返回明确业务错误

成功后返回最新用户资料，方便前端同步更新本地状态。

### 3. 上传头像

新增接口：

- `POST /api/v1/users/me/avatar`

请求类型：

- `multipart/form-data`

字段：

- `file`

行为约束：

- 仅接收图片文件
- 限制文件大小
- 上传成功后更新 `users.avatar`
- 返回最新头像地址或最新用户资料

## 文件存储方案

头像采用本地文件存储，作为当前阶段的最简实现。

建议存储位置：

- `backend/uploads/avatars/`

建议访问路径：

- `/uploads/avatars/{filename}`

文件命名建议：

- `userId-timestamp-random.ext`

这样可以避免：

- 原文件名冲突
- 特殊字符导致路径问题
- 用户上传同名文件互相覆盖

头像上传失败时不得写入数据库新地址，保持旧头像不变。

## 前端交互设计

在现有 [`Profile.vue`](/E:/项目文件/八股code/getoffer/src/views/Profile.vue) 页面内完成改造，不新增独立页面。

### 头像区域

在当前头像区域基础上新增：

- “更换头像”按钮
- 选择本地图片文件的上传入口
- 上传中的加载状态

交互流程：

1. 用户点击更换头像
2. 选择本地图片
3. 前端先校验格式和大小
4. 校验通过后立即上传
5. 上传成功后刷新头像显示并同步用户 store
6. 上传失败则提示错误并保留旧头像

### 个人信息区域

将当前只读的用户名展示调整为：

- 登录账号 `username`：只读展示
- 展示昵称 `nickname`：可编辑输入框

新增：

- “保存昵称”按钮
- 提交中禁用状态
- 表单级错误提示

展示规则：

- 页面头部名称优先显示 `nickname`
- 若 `nickname` 为空，则回退显示 `username`

### 导航与其他展示位

用户资料更新成功后，前端本地用户状态需要同步刷新，确保以下位置保持一致：

- 个人中心头部名称和头像
- 导航栏用户信息
- 其他依赖当前用户资料的展示位

## 错误处理

前端和后端都需要对错误做清晰反馈。

昵称修改：

- 为空：前端直接提示
- 超过限制：前端直接提示，后端兜底校验
- 已存在：后端返回明确错误信息，前端提示“昵称已被占用”
- 未登录：沿用现有鉴权失败处理

头像上传：

- 文件不是图片：前端拦截，后端兜底
- 文件超出大小限制：前端拦截，后端兜底
- 上传失败：提示失败信息，不覆盖旧头像
- 文件保存成功但数据库更新失败：应清理新文件或避免脏数据残留

## 后端实现建议

后端需要补充以下内容：

- `User` 实体新增 `nickname`
- 用户资料 DTO 返回 `nickname`
- 用户资料更新请求 DTO
- 头像上传响应 DTO 或统一返回用户资料 DTO
- `UserController` 新增资料修改与头像上传接口
- 用户资料服务层，集中处理昵称唯一性校验与头像更新逻辑
- 静态资源映射，暴露头像访问路径
- 全局异常处理补充可读错误信息

昵称唯一性校验应排除当前用户自己，避免用户保存原昵称时误判重复。

## 前端实现建议

前端需要补充以下内容：

- 用户资料类型映射新增 `nickname` 字段
- 用户 store 同步保存 `nickname` 和 `avatar`
- `authApi.me()` 返回结构兼容 `nickname`
- 新增用户资料更新 API
- 新增头像上传 API
- 个人中心页面表单与上传交互

为避免耦合，昵称修改与头像上传应使用两个独立请求，不合并成一个总保存接口。

## 测试范围

后端测试建议覆盖：

- 修改昵称成功
- 昵称重复时返回失败
- 提交原昵称时返回成功
- 头像上传成功
- 非图片文件上传失败
- 超限文件上传失败

前端测试建议覆盖：

- 资料页优先显示 `nickname`，为空时回退 `username`
- 昵称输入校验与错误提示
- 头像上传前校验
- 修改成功后本地用户状态同步更新

联调验证建议覆盖：

- 修改昵称后个人中心立即更新
- 修改昵称后导航栏立即更新
- 上传头像后页面立即显示新头像
- 刷新页面后昵称和头像仍能正确显示

## 实施顺序

建议按以下顺序实现：

1. 后端数据模型与 DTO 调整
2. 昵称修改接口
3. 头像上传与静态资源映射
4. 前端 API 与 store 更新
5. 个人中心页面交互改造
6. 测试补充与联调验证

## 风险与约束

- 本地文件存储适合开发和演示环境，不适合作为最终生产方案
- 头像目录需要加入忽略规则，避免上传文件进入版本管理
- 若数据库已有历史用户数据，需要明确 `nickname` 的默认值或允许为空
- 若后续作者名称展示改为 `nickname`，文章相关返回结构可能还需二次调整
