# MVP 后台管理系统设计

## 1. 背景

当前项目已经具备前台网站的核心能力，包括：

- 用户注册、登录、个人中心
- 文章发布、查看、收藏
- AI 会话创建、消息流式回复、历史记录

随着用户和内容逐步增加，单纯依赖数据库手工查看和修改已经不适合继续维护。因此需要补一个最小可用的后台管理系统，用于管理员登录、查看数据、审核内容，并为后续扩展权限体系打基础。

本次只做 MVP，不做复杂审批流、操作日志中心、多角色权限系统。

## 2. 目标

本次 MVP 后台需要满足以下目标：

- 管理员可通过独立入口 `/admin/login` 登录后台
- 后台与前台视觉和路由体系隔离
- 后台可查看核心统计数据
- 后台可查看用户列表
- 用户上传文章后，文章先进入后台待审核
- 管理员审核通过后，文章才在前台显示
- 管理员可审核拒绝文章
- 后台可查看 AI 会话列表
- 普通用户不能访问后台页面和后台接口

## 3. 范围

### 3.1 本次实现范围

- 后台登录页
- 后台布局页
- 数据概览页
- 用户管理页
- 文章审核管理页
- AI 会话管理页
- 管理员接口鉴权
- 用户角色字段与角色返回
- 文章审核状态字段与前台展示规则调整

### 3.2 本次不做

- 独立管理员表
- 多级角色体系
- 用户封禁/解封
- 编辑文章内容
- 用户自主重新提交审核
- 删除 AI 会话
- 操作审计日志
- 站点配置中心

## 4. 方案对比

### 方案 A：复用现有用户体系，在 `users` 表新增角色字段

做法：

- 在 `users` 表新增 `role`
- 继续复用现有 JWT 登录流程
- 前端通过 `/admin/login` 进入后台
- 后端新增 `/api/v1/admin/**` 接口并做管理员权限控制

优点：

- 改动集中
- 维护成本最低
- 最适合当前项目阶段

缺点：

- 当前只有 `USER` 和 `ADMIN` 两种角色，后续扩展时需要继续演进

### 方案 B：单独建立管理员表和管理员登录体系

优点：

- 账号体系边界绝对清晰

缺点：

- 登录、鉴权、维护成本明显增加
- 不适合当前 MVP 阶段

### 结论

采用方案 A。

## 5. 信息架构

### 5.1 后台入口

- `/admin/login`：管理员登录页
- `/admin`：后台根路径，登录后默认跳转 `/admin/dashboard`

### 5.2 后台页面

- `/admin/dashboard`：数据概览
- `/admin/users`：用户管理
- `/admin/articles`：文章审核管理
- `/admin/ai-sessions`：AI 会话管理

### 5.3 前台入口策略

- 前台页面不显示“后台管理”入口
- 管理员只能手动通过 `/admin/login` 进入后台

## 6. 数据模型设计

### 6.1 users 表变更

在 `users` 表新增：

- `role VARCHAR(20) NOT NULL DEFAULT 'USER'`

角色取值：

- `USER`
- `ADMIN`

说明：

- 现有普通用户默认都是 `USER`
- 管理员账号通过数据库手动改为 `ADMIN`

### 6.2 articles 表变更

在 `articles` 表新增：

- `status VARCHAR(20) NOT NULL DEFAULT 'PENDING'`

状态取值：

- `PENDING`
- `APPROVED`
- `REJECTED`

状态规则：

- 用户新上传文章时默认是 `PENDING`
- 管理员审核通过后变为 `APPROVED`
- 管理员审核拒绝后变为 `REJECTED`
- 只有 `APPROVED` 的文章可以在前台公开展示

### 6.3 实体变更

- `User` 实体新增 `role` 字段，并参与鉴权和响应返回
- `Article` 实体新增 `status` 字段，并参与前台查询与后台审核

## 7. 鉴权与权限设计

### 7.1 登录机制

继续复用现有登录接口：

- `POST /api/v1/auth/login`

登录成功后返回的用户信息中新增 `role`。

### 7.2 鉴权策略

后端接口分层：

- 普通前台接口：保持现状
- 后台接口：统一挂在 `/api/v1/admin/**`

访问控制规则：

- 未登录用户：不能访问 `/admin/**` 页面和 `/api/v1/admin/**` 接口
- 已登录但 `role !== 'ADMIN'`：不能访问后台页面和后台接口
- `role === 'ADMIN'`：可访问后台

### 7.3 前端路由守卫

- 访问 `/admin/login`
  - 若当前已登录且角色为 `ADMIN`，直接跳转 `/admin/dashboard`
- 访问 `/admin/**`
  - 未登录：跳转 `/admin/login`
  - 非管理员：跳转首页并提示无权限

### 7.4 后端权限校验

后端必须做强校验，不能只依赖前端路由守卫。

推荐做法：

- 在服务层或控制器层统一校验当前用户角色是否为 `ADMIN`
- 非管理员访问时返回明确错误

## 8. 文章审核流设计

### 8.1 用户发文流程

- 用户提交文章
- 后端创建文章，状态设为 `PENDING`
- 文章不会立即出现在前台公开列表

### 8.2 管理员审核通过

- 管理员在后台将文章状态改为 `APPROVED`
- 文章开始出现在首页、文章列表、文章详情等前台公开位置

### 8.3 管理员审核拒绝

- 管理员在后台将文章状态改为 `REJECTED`
- 文章不在前台公开展示
- 数据继续保留，便于后台追踪和后续扩展

### 8.4 前台展示规则

- 首页文章：只展示 `APPROVED`
- 文章列表：只展示 `APPROVED`
- 热门文章：只统计和展示 `APPROVED`
- 文章详情：普通用户只能访问 `APPROVED`
- 用户个人中心“我的文章”：可看到自己的全部文章，并带状态标签

## 9. 后台页面设计

### 9.1 后台布局

后台使用独立布局，不复用前台 `Navbar`。

布局结构：

- 左侧菜单
- 右侧内容区域

菜单项：

- 数据概览
- 用户管理
- 文章审核
- AI 会话

设计原则：

- 界面简洁
- 以实用和清晰为主
- 不追求复杂交互

### 9.2 数据概览页

展示 3 张核心统计卡片：

- 用户总数
- 文章总数
- AI 会话总数

本页不做图表，不做趋势分析。

### 9.3 用户管理页

功能：

- 分页查看用户列表
- 关键字搜索

建议显示字段：

- 用户 ID
- 用户名
- 昵称
- 手机号
- 角色
- 注册时间

本期不做编辑和删除用户。

### 9.4 文章审核管理页

功能：

- 分页查看文章列表
- 按状态筛选：全部、待审核、已通过、已拒绝
- 按标题或分类搜索
- 审核通过
- 审核拒绝
- 可选删除文章

建议显示字段：

- 文章 ID
- 标题
- 分类
- 作者
- 当前状态
- 发布时间

交互规则：

- 页面默认展示 `PENDING`
- 对待审核文章显示“通过”“拒绝”按钮
- 删除操作必须二次确认

### 9.5 AI 会话管理页

功能：

- 分页查看 AI 会话列表
- 支持基础搜索或按标题筛选

建议显示字段：

- 会话 ID
- 会话标题
- 所属用户
- 创建时间
- 更新时间

本期不做会话删除和消息详情查看。

## 10. 后端接口设计

### 10.1 管理后台数据概览

- `GET /api/v1/admin/dashboard`

返回：

- `userCount`
- `articleCount`
- `aiSessionCount`

### 10.2 用户管理

- `GET /api/v1/admin/users`

参数：

- `page`
- `pageSize`
- `keyword`

### 10.3 文章审核管理

- `GET /api/v1/admin/articles`

参数：

- `page`
- `pageSize`
- `keyword`
- `category`
- `status`

- `PATCH /api/v1/admin/articles/{articleId}/approve`

- `PATCH /api/v1/admin/articles/{articleId}/reject`

- `DELETE /api/v1/admin/articles/{articleId}`

### 10.4 AI 会话管理

- `GET /api/v1/admin/ai/sessions`

参数：

- `page`
- `pageSize`
- `keyword`

## 11. 现有前台接口调整

### 11.1 发文接口

- `POST /api/v1/articles`

行为调整：

- 保持用户调用方式不变
- 后端创建文章时默认写入 `status = PENDING`

### 11.2 文章公开查询接口

以下查询只返回 `APPROVED`：

- 文章列表
- 热门文章
- 文章详情

### 11.3 个人中心文章接口

- `GET /api/v1/users/me/articles`

行为调整：

- 继续返回当前用户自己的文章
- 每条文章增加 `status`
- 前端个人中心显示状态标签

## 12. 前端实现策略

### 12.1 路由结构

新增后台路由模块：

- `admin-login`
- `admin-layout`
- `admin-dashboard`
- `admin-users`
- `admin-articles`
- `admin-ai-sessions`

### 12.2 状态管理

继续复用现有用户登录态 store，但需要确保当前用户信息中包含 `role`。

后台页面判断管理员身份时，统一从 store 读取当前用户角色。

### 12.3 接口层

在现有 `frontend` API 模块基础上新增后台管理 API 封装，避免把后台请求和前台请求混在页面里。

### 12.4 前台文章状态展示

在个人中心“我的文章”中显示文章状态标签：

- `PENDING`：待审核
- `APPROVED`：已通过
- `REJECTED`：已拒绝

## 13. 错误处理

### 13.1 登录态问题

- token 失效：跳回 `/admin/login`
- 未登录：跳回 `/admin/login`

### 13.2 权限问题

- 普通用户进入后台：跳首页并提示“无权限访问后台”
- 非管理员调用后台接口：后端返回拒绝信息

### 13.3 页面加载失败

- 列表页显示错误提示
- 提供重新加载能力

### 13.4 审核失败

- 显示明确错误提示
- 保持当前列表状态不被破坏

## 14. 测试范围

### 14.1 后端测试

- 管理员可访问后台接口
- 普通用户不可访问后台接口
- 数据概览接口返回正确结构
- 用户列表接口可分页和搜索
- 用户发文后默认状态为 `PENDING`
- 管理员审核通过后文章状态变为 `APPROVED`
- 管理员审核拒绝后文章状态变为 `REJECTED`
- 前台公开文章查询只返回 `APPROVED`

### 14.2 前端测试

- 后台路由守卫生效
- 管理员登录后跳转后台首页
- 非管理员访问后台时被拦截
- 后台页面基础渲染正常
- 文章审核通过/拒绝操作后列表刷新正常
- 用户个人中心正确展示文章状态

## 15. 实施顺序建议

建议按以下顺序实现：

1. 数据模型补 `role` 与文章 `status`
2. 登录返回与用户信息补 `role`
3. 发文逻辑默认写入 `PENDING`
4. 前台公开文章查询改成只查 `APPROVED`
5. 后端管理员权限校验与后台接口
6. 后台路由与登录页
7. 后台布局与数据概览页
8. 用户管理页
9. 文章审核页
10. AI 会话管理页
11. 个人中心文章状态展示
12. 回归测试与联调

## 16. 风险与约束

### 风险 1：当前系统没有完整角色体系

影响：

- 本期只能支撑 `USER` / `ADMIN` 两类角色

处理：

- MVP 接受这个限制，后续再扩展

### 风险 2：管理员账号初始化依赖手工操作

影响：

- 第一个管理员需要手工指定

处理：

- 本期通过数据库手动设置管理员角色

### 风险 3：审核流增加了文章状态复杂度

影响：

- 前台文章查询、个人中心、后台审核都要同步调整

处理：

- 本次在 spec 中明确前后端展示规则，避免实现时理解不一致

## 17. 结论

本次 MVP 后台采用“独立后台入口 + 复用现有用户体系 + 新增角色字段 + 新增文章审核状态”的方案。

该方案能以较低复杂度补齐项目的后台管理能力，并引入最基础的内容审核流程，适合当前项目阶段。
