# MVP 后台管理系统设计

## 1. 背景

当前项目已经具备前台网站的核心能力，包括：

- 用户注册、登录、个人中心
- 文章发布、查看、收藏
- AI 会话创建、消息流式回复、历史记录

随着用户和内容逐步增加，单纯依赖数据库手工查看和修改已经不适合继续维护。因此需要补一个最小可用的后台管理系统，用于管理员登录、查看数据、管理内容，并为后续扩展权限体系打基础。

本次只做 MVP，不做复杂审批流、操作日志中心、多角色权限系统。

## 2. 目标

本次 MVP 后台需要满足以下目标：

- 管理员可通过独立入口 `/admin/login` 登录后台
- 后台与前台视觉和路由体系隔离
- 后台可查看核心统计数据
- 后台可查看用户列表
- 后台可查看并删除文章
- 后台可查看 AI 会话列表
- 普通用户不能访问后台页面和后台接口

## 3. 范围

### 3.1 本次实现范围

- 后台登录页
- 后台布局页
- 数据概览页
- 用户管理页
- 文章管理页
- AI 会话管理页
- 管理员接口鉴权
- 用户角色字段与角色返回

### 3.2 本次不做

- 独立管理员表
- 多级角色体系
- 用户封禁/解封
- 编辑文章内容
- 删除 AI 会话
- 操作审计日志
- 站点配置中心

## 4. 方案对比

### 方案 A：复用现有用户体系，在 `users` 表新增角色字段

做法：

- 在 `users` 表新增 `role`
- 继续复用现有 JWT 登录流程
- 前端通过 `/admin/login` 进入后台
- 后端新增 `/api/v1/admin/**` 接口并做管理员权限控制

优点：

- 改动集中
- 维护成本最低
- 最适合当前项目阶段

缺点：

- 当前只有 `USER` 和 `ADMIN` 两种角色，后续扩展时需要继续演进

### 方案 B：单独建立管理员表和管理员登录体系

优点：

- 账号体系边界绝对清晰

缺点：

- 登录、鉴权、维护成本明显增加
- 不适合当前 MVP 阶段

### 结论

采用方案 A。

## 5. 信息架构

### 5.1 后台入口

- `/admin/login`：管理员登录页
- `/admin`：后台根路径，登录后默认跳转 `/admin/dashboard`

### 5.2 后台页面

- `/admin/dashboard`：数据概览
- `/admin/users`：用户管理
- `/admin/articles`：文章管理
- `/admin/ai-sessions`：AI 会话管理

### 5.3 前台入口策略

- 前台页面不显示“后台管理”入口
- 管理员只能手动通过 `/admin/login` 进入后台

## 6. 数据模型设计

### 6.1 users 表变更

在 `users` 表新增：

- `role VARCHAR(20) NOT NULL DEFAULT 'USER'`

角色取值：

- `USER`
- `ADMIN`

说明：

- 现有普通用户默认都是 `USER`
- 管理员账号通过数据库手动改为 `ADMIN`

### 6.2 User 实体变更

`User` 实体新增 `role` 字段，并参与鉴权和响应返回。

## 7. 鉴权与权限设计

### 7.1 登录机制

继续复用现有登录接口：

- `POST /api/v1/auth/login`

登录成功后返回的用户信息中新增 `role`。

### 7.2 鉴权策略

后端接口分层：

- 普通前台接口：保持现状
- 后台接口：统一挂在 `/api/v1/admin/**`

访问控制规则：

- 未登录用户：不能访问 `/admin/**` 页面和 `/api/v1/admin/**` 接口
- 已登录但 `role !== 'ADMIN'`：不能访问后台页面和后台接口
- `role === 'ADMIN'`：可访问后台

### 7.3 前端路由守卫

- 访问 `/admin/login`
  - 若当前已登录且角色为 `ADMIN`，直接跳转 `/admin/dashboard`
- 访问 `/admin/**`
  - 未登录：跳转 `/admin/login`
  - 非管理员：跳转首页并提示无权限

### 7.4 后端权限校验

后端必须做强校验，不能只依赖前端路由守卫。

推荐做法：

- 在服务层或控制器层统一校验当前用户角色是否为 `ADMIN`
- 非管理员访问时返回明确错误

## 8. 后台页面设计

### 8.1 后台布局

后台使用独立布局，不复用前台 `Navbar`。

布局结构：

- 左侧菜单
- 右侧内容区域

菜单项：

- 数据概览
- 用户管理
- 文章管理
- AI 会话

设计原则：

- 界面简洁
- 以实用和清晰为主
- 不追求复杂交互

### 8.2 数据概览页

展示 3 张核心统计卡片：

- 用户总数
- 文章总数
- AI 会话总数

本页不做图表，不做趋势分析。

### 8.3 用户管理页

功能：

- 分页查看用户列表
- 关键字搜索

建议显示字段：

- 用户 ID
- 用户名
- 昵称
- 手机号
- 角色
- 注册时间

本期不做编辑和删除用户。

### 8.4 文章管理页

功能：

- 分页查看文章列表
- 按标题或分类搜索
- 删除文章

建议显示字段：

- 文章 ID
- 标题
- 分类
- 作者
- 发布时间

删除操作要求：

- 必须二次确认
- 删除成功后刷新列表

### 8.5 AI 会话管理页

功能：

- 分页查看 AI 会话列表
- 支持基础搜索或按标题筛选

建议显示字段：

- 会话 ID
- 会话标题
- 所属用户
- 创建时间
- 更新时间

本期不做会话删除和消息详情查看。

## 9. 后端接口设计

### 9.1 管理后台数据概览

- `GET /api/v1/admin/dashboard`

返回：

- `userCount`
- `articleCount`
- `aiSessionCount`

### 9.2 用户管理

- `GET /api/v1/admin/users`

参数：

- `page`
- `pageSize`
- `keyword`

### 9.3 文章管理

- `GET /api/v1/admin/articles`

参数：

- `page`
- `pageSize`
- `keyword`
- `category`

- `DELETE /api/v1/admin/articles/{articleId}`

### 9.4 AI 会话管理

- `GET /api/v1/admin/ai/sessions`

参数：

- `page`
- `pageSize`
- `keyword`

## 10. 前端实现策略

### 10.1 路由结构

新增后台路由模块：

- `admin-login`
- `admin-layout`
- `admin-dashboard`
- `admin-users`
- `admin-articles`
- `admin-ai-sessions`

### 10.2 状态管理

继续复用现有用户登录态 store，但需要确保当前用户信息中包含 `role`。

后台页面判断管理员身份时，统一从 store 读取当前用户角色。

### 10.3 接口层

在现有 `frontend` API 模块基础上新增后台管理 API 封装，避免把后台请求和前台请求混在页面里。

## 11. 错误处理

### 11.1 登录态问题

- token 失效：跳回 `/admin/login`
- 未登录：跳回 `/admin/login`

### 11.2 权限问题

- 普通用户进入后台：跳首页并提示“无权限访问后台”
- 非管理员调用后台接口：后端返回拒绝信息

### 11.3 页面加载失败

- 列表页显示错误提示
- 提供重新加载能力

### 11.4 删除失败

- 弹出错误提示
- 保持当前列表状态不被破坏

## 12. 测试范围

### 12.1 后端测试

- 管理员可访问后台接口
- 普通用户不可访问后台接口
- 数据概览接口返回正确结构
- 用户列表接口可分页和搜索
- 文章删除接口可执行并返回成功

### 12.2 前端测试

- 后台路由守卫生效
- 管理员登录后跳转后台首页
- 非管理员访问后台时被拦截
- 后台页面基础渲染正常
- 删除文章的确认交互和列表刷新正常

## 13. 实施顺序建议

建议按以下顺序实现：

1. 数据模型与登录返回补 `role`
2. 后端管理员权限校验与后台接口
3. 后台路由与登录页
4. 后台布局与数据概览页
5. 用户管理页
6. 文章管理页
7. AI 会话管理页
8. 回归测试与联调

## 14. 风险与约束

### 风险 1：当前系统没有完整角色体系

影响：

- 本期只能支撑 `USER` / `ADMIN` 两类角色

处理：

- MVP 接受这个限制，后续再扩展

### 风险 2：管理员账号初始化依赖手工操作

影响：

- 第一个管理员需要手工指定

处理：

- 本期通过数据库手动设置管理员角色

### 风险 3：后台能力先偏查看和删除

影响：

- 后续若需要更多内容治理能力，还要继续扩展

处理：

- 先确保 MVP 跑通，再迭代

## 15. 结论

本次 MVP 后台采用“独立后台入口 + 复用现有用户体系 + 新增角色字段”的方案。

该方案能以最低复杂度补齐项目的后台管理能力，同时保持和现有前后端架构一致，适合当前项目阶段。
